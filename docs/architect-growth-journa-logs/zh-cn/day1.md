### 📅 Day 1: 基础设施奠基日

#### 🛠️ 今天我做了什么

1. **GitHub 仓库工程化初始化**：在本地和 GitHub 初始化了 `Project-OmniGuard` 仓库。建立了符合企业级零信任纪律的目录矩阵（包含 `.azure/`, `src/client-edge/`, `src/cloud-orchestrator/`, `docs/adrs/`）。
2. **手写首版 IaC 脚本**：纯手工编写了第一版 `.azure/main.bicep` 基础设施即代码脚本，一键自动化创建了符合新加坡金融合规标准的 Hub-Spoke 双虚拟网络（`omni-hub-vnet` 与 `omni-spoke-vnet`）。
3. **建立内网对等互联**：在 Hub 网络与 Spoke 网络之间配置了双向 VNet Peering 隧道，打通了跨网段内网大动脉。确保数据包在云端内部骨干网直接转发，彻底拒绝暴露在公网互联网。
4. **FinOps 生命周期验证**：遵循成本控制纪律，通过 Azure CLI 运行全量部署测试。在图形界面和命令行查验网络控制面状态成功后，立刻通过脚本一键物理销毁测试资源组，达成零残留、零财务波动的开发闭环。

#### 🎯 为什么我这么做（架构权衡与反思）

* **工程确定性 对抗 手动配置漂移**：作为云解决方案架构师，安全感来源于“可重现性”。我完全放弃了在 Azure Portal 网页端纯手动点击创建资源的低效做法，坚持 100% IaC 代码驱动。手动操作无法留下审计痕迹，且易导致配置漂移；而代码是系统唯一的信任源。
* **金融级网络底座的必要性**：市场上绝大多数 AI “玩具应用”都直接裸奔在带有公网 IP 的不安全网段。引入 Hub-Spoke 双虚网架构是为了在系统底层确立明确的防线：Hub 区用于未来托管审计、集中防火墙等安全策略，Spoke 区作为锁死 AI 运行时和隐私数据的隔离监牢，从第一天起就从拓扑上消灭数据泄露隐患。
* **极致预算工程 (FinOps 护栏)**：面对不到 90 美金的紧张 Azure 订阅额度，我必须践行“神形分离”的肌肉记忆。代码（形）可以天天写、天天 Push；但云端实体（神）只允许在测试验证的 5 分钟内活过来，测试完立马物理蒸发（`az group delete`），把财务成本作为架构设计的核心约束条件。

### 📅 Day 2: 云上治理与微隔离日

#### 🛠️ 今天我做了什么

1. **订阅级架构提维**：为了达成“完全自动化”的目标，将 `main.bicep` 的作用域强制提升至整个订阅高度（`targetScope = 'subscription'`）。把 `omni-guard-rg` 资源组本身的创建、标签规范（Tags）和生命周期管理完全锁进代码流水线。
2. **数据与基础设施控制面解耦**：放弃在云模板中硬编码防火墙规则。将具体的细颗粒度网络安全策略完全隔离到外部的纯 JSON 矩阵文件 `.azure/network-rules.json` 中，并在 Bicep 主模板中利用高杠杆的 `json(loadTextContent())` 语法动态加载与运行时解析。
3. **子网细颗粒度微隔离矩阵**：手工编写了零信任网络安全组（NSG）规则链。通过 `omni-backend-nsg` 默认格杀一切来自 Internet 的公网入站流量；通过 `omni-storage-nsg` 确立严苛的安全纪律——在物理上只放行来自 Backend 业务子网（`10.1.1.0/24`）的 443 端口内网握手，对其余任何越界流量格杀勿论（Deny）。
4. **多层级组件解耦 (Nested Module)**：落地了多级模板嵌套设计。通过建立独立的 `.azure/nested-infra.bicep` 特种部队文件，利用 `module` 语法将降落场上下文切换（Scope Down）至目标资源组内部，实现了高内聚、职责分离的参数 props 投递。

#### 🎯 为什么我这么做（架构权衡与反思）

* **指挥官与特种部队的联动（作用域冲突遭遇战）**：在真实开荒中，遭遇了平台的硬核物理限制——站在订阅高度的单脚本文件无法直接声明创建资源组内部的子网细节。我借鉴了 10 年前端复杂系统开发中“顶层状态/路由与子组件解耦”的组件化思维，将代码切分为 `main`（指挥官）与 `nested`（子模块）。大局观拉满，优雅打破平台限制。
* **配置驱动的高杠杆治理**：将安全阻断规则锁死在云脚本中会导致系统极难维护。将其抽离为纯 JSON 矩阵，完美向跨国跨文化大厂的工程纪律看齐：未来外部合规审计员或首席安全官（CSO）只需单独迭代 JSON 规则矩阵，基础设施底座代码无需任何变更，实现了真正的高杠杆。
* **物理激活特性的深度自证**：在物理自证环节，我发现若子网内未挂载任何活动网卡（NIC），Azure 网络的底层路由表不会被真正实例化，Portal 网页端的有效路由表菜单会被强行隐藏。我坚决不通过盲目拉起昂贵的大型 VM 去浪费额度，而是运用高级 SA 的 CLI 深度审计手段（`az network nsg show`），跨过网页表象，直接对底层控制面的安全载荷（Payload）进行无死角检索。零成本、绝对干净地达成了 100% 确定性的安全自证。
