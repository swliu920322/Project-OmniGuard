# 📑 Project-OmniGuard: Cloud AI Architect Growth Journal

### 📅 Day 2: 云上治理与微隔离日

#### 🛠️ 今天我做了什么

1. **订阅级架构提维**：为了达成“完全自动化”的目标，将 `main.bicep` 的作用域强制提升至整个订阅高度（
   `targetScope = 'subscription'`）。把 `omni-guard-rg` 资源组本身的创建、标签规范（Tags）和生命周期管理完全锁进代码流水线。
2. **数据与基础设施控制面解耦**：放弃在云模板中硬编码防火墙规则。将具体的细颗粒度网络安全策略完全隔离到外部的纯 JSON 矩阵文件
   `.azure/network-rules.json` 中，并在 Bicep 主模板中利用高杠杆的 `json(loadTextContent())` 语法动态加载与运行时解析。
3. **子网细颗粒度微隔离矩阵**：手工编写了零信任网络安全组（NSG）规则链。通过 `omni-backend-nsg` 默认格杀一切来自 Internet
   的公网入站流量；通过 `omni-storage-nsg` 确立严苛的安全纪律——在物理上只放行来自 Backend 业务子网（`10.1.1.0/24`）的 443
   端口内网握手，对其余任何越界流量格杀勿论（Deny）。
4. **多层级组件解耦 (Nested Module)**：落地了多级模板嵌套设计。通过建立独立的 `.azure/nested-infra.bicep` 特种部队文件，利用
   `module` 语法将降落场上下文切换（Scope Down）至目标资源组内部，实现了高内聚、职责分离的参数 props 投递。

#### 🎯 为什么我这么做（架构权衡与反思）

* **指挥官与特种部队的联动（作用域冲突遭遇战）**：在真实开荒中，遭遇了平台的硬核物理限制——站在订阅高度的单脚本文件无法直接声明创建资源组内部的子网细节。我借鉴了
  10 年前端复杂系统开发中“顶层状态/路由与子组件解耦”的组件化思维，将代码切分为 `main`（指挥官）与 `nested`
  （子模块）。大局观拉满，优雅打破平台限制。
* **配置驱动的高杠杆治理**：将安全阻断规则锁死在云脚本中会导致系统极难维护。将其抽离为纯 JSON
  矩阵，完美向跨国跨文化大厂的工程纪律看齐：未来外部合规审计员或首席安全官（CSO）只需单独迭代 JSON
  规则矩阵，基础设施底座代码无需任何变更，实现了真正的高杠杆。
* **物理激活特性的深度自证**：在物理自证环节，我发现若子网内未挂载任何活动网卡（NIC），Azure 网络的底层路由表不会被真正实例化，Portal
  网页端的有效路由表菜单会被强行隐藏。我坚决不通过盲目拉起昂贵的大型 VM 去浪费额度，而是运用高级 SA 的 CLI 深度审计手段（
  `az network nsg show`），跨过网页表象，直接对底层控制面的安全载荷（Payload）进行无死角检索。零成本、绝对干净地达成了 100%
  确定性的安全自证。
